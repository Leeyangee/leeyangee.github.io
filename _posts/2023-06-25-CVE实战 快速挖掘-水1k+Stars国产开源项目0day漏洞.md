# [](#header-1)0、前言
由于src挖多了，每天盯着那一堆框架看很累，挖不到洞更累，隧决定去github上找开源项目挖凑凑cve  
然后发现某国产开源web项目，近期closed的issue比较少，就决定挖它了  

# [](#header-1)1、快速资产整理

先看看repository里以往closed的issue，看一下大佬们挖掘思路，再看看该项目哪些地方可能经常存在问题  
发现几个月前某安全实验室对该项目进行大量测试，提交了大量关于系统、角色类controller的sql注入及敏感rce，在这种情况下，想分得一杯残羹剩饭很不容易  
再已知在这几个月时间内没有新功能被加入，在这类情况下，黑盒为主、白盒辅助必然成了主要测试方式，也主要关照xss、文件上传、未授权访问等此类漏洞点  

正式搭建项目，该项目采用的是基于ThinkPHP6.1框架延伸的经典全栈式的mvc模型，相信各位是很熟悉了的  
安装时只需配置数据库用户名、密码即可一键自动安装，部署非常快速方便  
我们先看后台部分，后台部分分为经典的以下几个板块  
```
系统设置
系统日志、黑名单...balabala
用户角色管理、用户管理
后台用户角色管理、后台用户管理
插件管理...balabala
```


# [](#header-1)2、上手
在简单的整理并对一些敏感点排除危害后，发现插件管理处可以离线上传插件  
大概意思就是，可以从官网上下载插件压缩包，再通过离线上传这个功能导入我们下载的压缩包  
下载了一个官方插件并解压，插件中大致路径是这样的：  
```
--README.md
--public(插件前端部分)
----img
----js
----css
----index.html
--controller
----controller1.php
----controller2.php
```
在这里，我试了试修改controller中的代码，再把它打包成一个插件从离线上传处上传，发现居然能够上传成功？  

在第一次上传插件后，想删除当前插件并进行第二次上传时，我发现竟然怎么点删除都删除不了  
当时以为是有某种检测机制在运行，隧回到插件controller仔细阅读上传插件那部分的代码，我发现并没有问题，但就是删除不了  
后来只好把该项目又重新配置了一遍，这才得以继续上传  

再次在controller中添加payload部分，某个参数可触发该payload，再次打包上传，居然也能上传成功？！  
再再次在public中直接添加webshell payload，再次打包上传，发现蚁剑成功连接，居然又又又能上传成功？？！！  

测到这里我已经很惊讶了，一个1k+的半商业性开源项目居然会犯如此低级问题，连点过滤都没有那么顺利？？  

# [](#header-1)3、调查
经过调查发现，其实早在几个月前低版本时，就已经有人发现该漏洞并提交issue了，作者也积极修复了，但不知为何， 这个问题又在高版本中突然出现了  
经过合理的推测，我认为这个漏洞的造成原因应该是作者在合并分支时出现了错误，但也仅仅是一种推测而已  
再次一一查看一下已经closed的issues，结果是都修复了，该漏洞可能正好是我碰巧遇到的  

提交cve，在半个月以后成功分得编号，算是成功水了一个小cve吧  
由于涉及到隐私问题，cve编号不在此放出来
