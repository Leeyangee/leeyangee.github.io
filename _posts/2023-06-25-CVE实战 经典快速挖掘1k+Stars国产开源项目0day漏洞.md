# [](#header-1)0、前言
由于src挖多了，每天盯着那一堆框架看很累，隧决定去github上找开源项目挖挖  
然后发现某国产开源cms项目，近期closed的issue比较少，就决定挖它了  

# [](#header-1)1、快速资产整理
该项目采用的是基于ThinkPHP6.1框架延伸的经典全栈式mvc前后端分离框架，安装时只需配置数据库用户名、密码即可一键自动安装，非常方便  
既然是快速挖掘，那我们先看后台部分，后台部分分为经典的以下几个板块  
```
系统设置
系统日志、黑名单balabala
用户角色管理、用户管理
后台用户角色管理、后台用户管理
插件管理balabala
```
再看看往常提交的issue，发现两个月前某些关于系统、角色类的sql注入及敏感rce已经被大量提交  
再已知在这两个月时间内没有新功能被加入，想必在这种情况下，黑盒测试必然成了主要测试方式，也主要关照xss、文件上传、未授权访问等此类漏洞点  

# [](#header-1)2、上手
在简单的整理后，隧发现插件管理处可以离线上传插件  
大概意思就是，可以从官网上下载插件压缩包，再通过离线上传这个功能导入我们下载的压缩包  
下载了一个官方插件并解压，插件中大致路径是这样的：  
```
--README.md
--public(插件前端部分)
----img
----js
----css
----index.html
--controller
----controller1.php
----controller2.php
```
在这里，我试了试修改controller中的代码，发现居然能够上传成功？  
再次在controller中添加payload部分，某个参数可触发该payload，居然也能上传成功？！  
再再次在public中直接添加webshell payload，发现蚁剑成功连接，居然又又又能上传成功？？！！  

测到这里我已经很惊讶了，一个1k+的半商业性开源项目居然会犯如此低级问题  

# [](#header-1)3、调查
经过调查发现，其实早在几个月前低版本时，就已经有人发现该漏洞并提交issue了，作者也积极修复了，但不知为何， 这个问题又在高版本中出现了  
再次一一查看一下已经closed的issues，结果是基本上都修复了，该漏洞可能正好是我碰巧遇到的  

再次提交cve，在半个月以后成功分得编号，算是成功水了一个吧
