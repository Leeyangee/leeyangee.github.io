# [](#header-1)0、前言
这是一次本人漏洞挖掘实战过程

总体而言，比起其他站点七天往上的审核日期以及长久不修复的漏洞，上海交通大学处理漏洞的速度可以说是神速了
  
# [](#header-1)1、资产搜集阶段

在对前期浅浅地搜索发现，上海交通大学的资产面相当地广，资产间联系相当复杂.  光已知的域名(fofa搜集、domainbrute硬跑)就有几千个  
因此，对上交的测试策略我采用扫描为主，手动测试(包括扫描器扫出的漏洞)为辅助，逻辑漏洞就只测几个大站点和几个看起来容易出逻辑洞的站点.  

我常常使用以下几个资产搜集工具进行工作：

1、ARL资产搜集灯塔: 过时的大型神器，比较偏向于http(s)资产  
缺点是在桥接模式、NAT模式的虚拟机下以及低带宽(亲测至少20Mb及以上才准确)下无法正常工作. 
```
  我在issue上找了很久都没找到原因，最后在StackOverFlow上一篇文章里面找到了原因
  在桥接模式下，因为docker无法bind主机127.0.0.11:53而无法开启ARL dns解析服务.
  而ARL的站点搜集功能是必须开启该dns服务，因此ARL站点搜集服务将会停止，后续的fileleak、site_spider等特色功能也会停止
```
优点是自研的fileleak以及URL爬虫简直是神中神，以至于有一些资产搜集工具移植fileleak作为自己的插件
```
  在之前，我使用fileleak和xray做了对比，发现fileleak更能搜集到敏感的api资产和文件资产. 只能说各有各的优点
```

2、Fofa：神中神不多说，毕竟花钱了  
缺点就是对于个人用户一个月99太贵了，但是可以去咸鱼上买13块一个月的共享账号  

3、Goby：大型分布式搜集神器，偏向于基于tcp/udp的资产(比如像ssh、smtp)  
缺点就是无法像ARL一样做到完全自动整合自动扫描http资产，并且端口扫描时间过长，Bug太多  
```
  家庭10Mb带宽Goby扫C段全端口至少需要花上三天，因此我基本上不用Goby作全端口扫描
  并且新版Goby无法修改端口扫描速率，这个Bug长时间不修复
```

4、Google Hacking：也是神器，不建议用来搜集域名、IP等信息，搜集身份证、学生证、银行卡等资产很方便
