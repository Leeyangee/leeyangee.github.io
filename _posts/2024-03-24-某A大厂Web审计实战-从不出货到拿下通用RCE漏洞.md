# [](#header-1)0x01、资产发现过程  

想拿一个大厂通用RCE还是一个比较难的事情，毕竟拿RCE直接能进服务器内网. 这要求攻击者具有较高资产搜集和代码审计能力，并且该资产相对于其他资产比较脆弱.  

因此在这里，挑选合理资产的方法就成为了重中之重  

在这里，笔者的初步筛选方法是:  
1. 先初步统计有哪些白盒应用，再分类哪些是开源的、源代码公开/泄露的，方便后续审计工作.  
2. 而后有源码系统的，直接github/全局查找语法搜索RCE构造点，如果搜到能RCE的危险函数就尽量想方设法向上利用.  
3. 如果实在发现不了RCE，那就退而求其次找文件上传和其他中危漏洞. 在特殊情况下可以升级为RCE.  

# [](#header-1)0x02、资产锁定

经过两天三夜的排查后，终于发现一个比较可疑的Web应用，该Web应用在互联网中的资产十分少，但却具有惊人的通用性  
(此处的特征与其他应用重合，真实IP条数没有那么多)  

![avatar](/image/2024-03-23-0.png)

都是Web应用，为何我偏偏会注意到该应用呢？因为  
1. 经过资产测绘发现，该应用使用了较为简单的Py构建. 在我个人的刻板印象中使用Py构建Web应用的大多为数据科学家/测试工程师，这些岗位对Py特性不是很了解，易导致RCE  
2. 该应用页面构造简单，有可能在参数序列化方面使用了较为危险的函数，比较容易导致RCE  

# [](#header-1)0x03、源码搜集

经过不断搜集，笔者终于发现了在某个较为远古的该大厂开源仓库的commits中遗留下来的该Web应用的身影.  
好巧不巧该厂只是把该应用的分支开源版本作为补充放到了一个开源仓库的路径下，要是放在一个独立仓库里那笔者的搜集阶段估计会被大大拉长了  

在此笔者通过遗留的commits信息得到了该应用的部分开源版本2.0的代码:  
![avatar](/image/2024-03-23-1.png)

在README中，确实印证了我一开始的刻板印象: 该应用确实是一个分布式测试框架  
![avatar](/image/2024-03-23-2.png)

至此，通过观察代码，笔者大概了解了该Web应用的来龙去脉:

# [](#header-1)0x04、正式审计

首先，该框架的大致原理是:  
Web接口服务浅处理数据 -> 分布式处理服务深处理数据  

由于在此懒得配置分布式环境，因此先浅测一下Web接口服务.  
首先，Web接口相对于其他应用是比较简单的. 就是一个Python原生web.py + 自定义传参. 到这里其实工程师的水平也差不多摸透了  
![avatar](/image/2024-03-23-3.png)

逐个查看Handler，感觉写的中规中矩没什么特别能getshell的地方.  
本来差点就想部署分布式去测了，但在临走之前鬼使神差地看了眼加载静态页面如下Handler，没曾想直接出货了  
![avatar](/image/2024-03-23-5.png)  
首先，笔者对如上代码进行解释:  
name是存放的值是从客户端传回的html模板，例如: http://127.0.0.1/index.html的name值就为index  
r是web.py加载的模板，例如: r.index()该函数返回index页面的html  
但在此，工程师却使用如下代码实例化函数加载html  
```py
func = eval('r.'+name)
return func()
```
明明可以使用getattr等反射中更加安全的办法，工程师却偏偏使用了大名鼎鼎的eval，相信该函数的危害笔者在此不过多强调读者也知.  

接下来顺藤摸瓜根据利用链构造Poc，如果不出意外在路径中构造  
```py
http://127.0.0.1/index+__import__('os').system('calc').html
```
那name变量的值就会变成  
```py
name = "r.index+__import__('os').system('calc')"
```
就直接能触发命令执行  

# [](#header-1)0x05、漏洞复现

笔者在本地8001端口上搭建了该Web服务，并访问  
```url
http://127.0.0.1:8001/index+__import__('os').system('calc').html
```

服务端直接弹出计算器，证明存在RCE漏洞
